# RISKS — qa-autopilot

운영형 QA 자동화의 리스크 레지스터. 심각도(S)와 가능성(L)을 1~5로, 점수 = S×L.
각 리스크는 **완화책 + 검증 지표 + 트리거(언제 멈출지)**를 함께 적는다.

> 자동화 도구가 잘못 동작하면 "사람이 직접 하던 것보다 더 나쁜" 결과(잘못된 수정의 자동 확산, 진짜 버그의 노이즈 매몰)를 낳는다. 그래서 리스크 관리가 곧 제품이다.

---

## R1. AI 분류 오버트러스트 (S5 × L4 = 20) 🔴

**증상:** Triage가 진짜 PRODUCT_BUG를 FLAKY로 잘못 분류 → Quarantine으로 격리 → 실버그가 릴리즈를 통과.

- **완화:**
  - confidence 임계값(`triage.confidenceThreshold`) 미만은 무조건 Human Queue.
  - 분류 정확도 자체를 **메타 지표**로 추적(사람이 뒤집은 비율 = override rate).
  - Quarantine 항목도 주기적 재평가(영구 격리 금지, TTL).
- **검증 지표:** override rate < 10%, quarantine에서 사후 PRODUCT_BUG로 재분류된 건수 = 0 목표.
- **STOP 트리거:** override rate가 25% 초과하면 자동 라우팅 끄고 전건 Human Queue로 강등.

## R2. 잘못된 소스 수정의 자동 확산 (S5 × L3 = 15) 🔴

**증상:** AI가 그럴듯하지만 틀린 app_source 수정을 제안 → 사람이 충분히 안 보고 승인 → 회귀.

- **완화:**
  - app_source는 `autoPr: false` + 승인 2인 + CODEOWNERS 필수(`qa.config.yaml`).
  - **AI는 머지 권한 없음.** PR + 회귀 재실행 증빙까지만.
  - 검증 없는 수정은 게이트 통과 불가(`regression_proof` 필수).
  - 수정 영향 범위(diff가 건드린 파일/공개 API)를 PR에 자동 표기.
- **검증 지표:** 병합된 remediation PR의 사후 롤백률 < 5%.
- **STOP 트리거:** 롤백률 > 15%면 app_source 자동 제안 비활성, test_only만 유지.

## R3. 플레이키 무한 재시도 비용 폭주 (S4 × L4 = 16) 🔴

**증상:** Retry Lane이 통과할 때까지 재시도 → STT/LLM/GPU 재호출로 과금 폭주(Actnote Modal `retries=3` 전체 재과금 교훈).

- **완화:**
  - `retry.maxRetries` 하드 상한 + 백오프.
  - 재시도 대상 에러 타입 화이트리스트(`retryableErrors`)만. assertion 실패는 재시도 금지.
  - 실행당/사이클당 **비용·시간 예산 상한** 도입(Phase 1).
- **검증 지표:** 사이클당 재시도 총횟수, 재시도 유발 비용.
- **STOP 트리거:** 예산 상한 도달 시 즉시 중단하고 미실행분을 SKIPPED로 보고.

## R4. 노이즈가 신호를 매몰 (S4 × L3 = 12) 🟡

**증상:** flakySignals 키워드가 과하게 광범위 → 진짜 실패까지 Quarantine.

- **완화:**
  - Phase 1 규칙 기반은 보수적으로(좁게) 시작, AI Triage가 보강.
  - Quarantine 격리도 감사 로그에 남기고 대시보드로 가시화.
- **검증 지표:** quarantine 비율 추세, 격리 후 사람이 신호로 되돌린 비율.
- **STOP 트리거:** quarantine 비율 > 40%면 flakySignals 재검토.

## R5. caseId 불안정 → 클러스터링/추세 붕괴 (S3 × L4 = 12) 🟡

**증상:** 테스트 제목/파일 변경으로 `caseId`가 흔들리면 동일 실패가 매번 신규로 잡혀 추세·클러스터가 깨진다.

- **완화:** caseId는 파일+제목 기반의 안정 키. 리네임 추적(별칭 매핑) Phase 2 도입.
- **검증 지표:** 신규 caseId 출현율 vs 실제 테스트 추가 수의 괴리.

## R6. 정규형의 표현력 부족 (S3 × L3 = 9) 🟡

**증상:** 새 러너의 결과가 `TestResult`에 안 맞아 정보 손실 → 오분류.

- **완화:** `meta` 자유 필드 + 어댑터별 raw 보존(객체 스토리지). 정규형은 버전닝.
- **검증 지표:** 어댑터 추가 시 매핑 커버리지 리뷰.

## R7. 거버넌스 우회 / 권한 오남용 (S5 × L2 = 10) 🟡

**증상:** 자동 경로가 승인 게이트를 건너뛰거나, 봇 계정이 자기 PR을 자기가 승인.

- **완화:** append-only `audit_log`. 승인자는 항상 사람 식별자. self-approve 차단. 정책 스냅샷을 런과 함께 저장.
- **검증 지표:** 게이트 우회 감지 알람 0건.

## R8. 대상 레포 시크릿/PII 노출 (S5 × L2 = 10) 🟡

**증상:** 실패 로그/트레이스에 토큰·실제 회의 transcript(PII)가 섞여 AI 프롬프트나 대시보드로 유출(Actnote: 실제 transcript 사용 금지 규칙).

- **완화:** 아티팩트 수집 단계에서 secret/PII 마스킹. AI에 보내는 컨텍스트는 마스킹본만. 시크릿은 도구에 전달하지 않음.
- **검증 지표:** 마스킹 룰 적용 누락 스캔.
- **STOP 트리거:** PII 유출 1건이라도 확인되면 AI 컨텍스트 전송 즉시 중단.

## R9. 자동화에 대한 신뢰 상실 (S4 × L3 = 12) 🟡

**증상:** 오탐/잘못된 PR이 반복되면 팀이 도구를 무시하고 수동 회귀.

- **완화:** Phase별 점진 도입(분류→격리→거버넌스 신뢰 누적 후에야 수정 자동화). 모든 자동 판단에 rationale 노출.
- **검증 지표:** 제안 수용률, 사람이 도구 출력을 실제로 보는 비율.

## R10. 비결정적 LLM 출력의 통과/실패 이분법 부적합 (S3 × L3 = 9) 🟡

**증상:** Actnote 액션 개수 변동 같은 건 pass/fail이 아니라 추세·임계로 봐야 하는데 단순 단언으로 처리하면 노이즈 양산.

- **완화:** LLM eval은 별도 모듈(추세·분포·임계)로 분리. 현재 범위(E2E/pytest)에는 미포함, 백로그.

## 횡단 원칙

- **기본값은 항상 사람에게:** 애매하면 자동 처리하지 않는다.
- **모든 자동 행위는 audit_log에 불변 기록:** 사후 추적 가능해야 한다.
- **STOP 트리거를 미리 정의:** 지표가 임계를 넘으면 자동화를 단계적으로 끈다.

## STOP 트리거 구현 현황

R1·R2·R4의 STOP 트리거는 `@qa/metrics`에 코드화되어 CLI가 매 사이클 평가·적용한다:

| 위험 | 지표 | 임계 | 발동 시 제어 | 적용 위치 |
|---|---|---|---|---|
| R1 | overrideRate | > 25% | `forceHumanTriage` | `TriageEngine({forceHuman})` — 전건 사람 강등 |
| R2 | rollbackRate | > 15% | `disableAppSourceRemediation` | `remediate({disableAppSource})` — app_source 제안 생략 |
| R4 | quarantineRatio | > 40% | `reviewFlakySignals` | (경고) flakySignals 재검토 |

- 누계는 **CLI·콘솔 공유 스토어**(`DATABASE_URL` → `metrics` 테이블 / 아니면 `metrics.json`).
- **피드백 자동 캡처**: 콘솔의 "분류 이의"(R1)·"롤백 보고"(R2) 버튼 + 승인 정족수 충족 시 merged 자동 +1. `qa feedback`는 수동 폴백.
- `minSamples`(기본 20)로 소표본 오발 방지. **과거 성과가 현재 사이클을 게이팅**한다.
- R3(재시도 비용)는 별도로 `core/budget.ts`가 사이클 내에서 직접 차단.
- 한계: 콘솔 버튼의 쓰기(Server Action) HTTP 경로는 브라우저/E2E로 검증 필요. 읽기·평가 경로 및 기록 함수는 검증됨.
